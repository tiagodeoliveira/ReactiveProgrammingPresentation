Reactive Programming

  • Observer pattern
    • When a given object keeps a list of who is using it and notifies on every and each change
  • Object-oriented reactive programming
    • When the object attributes react to changes individually
  • Functional reactive programming
    • Based on the classic functional programming (mathematical functions evaluation) brings a notion of "over time" values, values that can change itself based on a function evaluation.
  • Dataflow programming
    • Defines a 'directed graph' of data flowing through the functions.
  • Stream -> Stream -> Stream
    • Put all those concepts together, now we have async streams everywhere of anything. Thats the core!!!!

Reactive Architecture

  • Dataflow Architecture
    • The opposite of control flow architecture (von Neumann)
    • All the flow is based on input args, what makes it a nondeterministic model
  • MVVM (Model View ViewModel)
    • Derived from MVC
    • The UI knows its own data model, and handle all view display logic
    • But never holds any business logic
  • Message Oriented Architecture
    • Creates an easy and powerful distributed architecture
  • Reactive Microservices
    • Small services, working as autonomous systems, reacting to events
  • Amazon AWS gives a huge help on elasticity and resilience
  • Message driven systems are essential to achieve elasticity and responsiveness
  • Non blocking processes allow to have a small and maybe fix response time, it means service quality

The Reactive Manifesto

  • Responsive
    • Focus on providing rapid and consistent response times, establishing reliable upper bounds to deliver a consistent QOS
  • Resilient
    • The system stays responsive in the face of failure, using: replication, containment, isolation and delegation
  • Elastic
    • The system stays responsive under varying workload
    • Can react to changes in the input rate by increasing or decreasing the resources allocated to service these inputs
  • Message Driven
    • Reactive Systems rely on asynchronous message-passing to establish a boundary between components that ensures loose coupling, isolation, location transparency, and provides the means to delegate errors as messages
